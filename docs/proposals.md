## 0. 判断の物差し（何に照らして選んだか）

提案はすべて、既存文書が定めた製品の意図に照らして選定・却下した。

1. **`.velq` の普及が最優先**（requirements §9-2）— フォーマットを「開ける人・作る場面」を増やすものは強い。
2. **差別化は 3 本柱** — ファイルマネージャ / 保存履歴 / `.velq`（CLAUDE.md）。柱を深める投資を優先。
3. **北極星「Finder 並みに分かりやすく、でも書く人のための」**（ui-ux-spec 冒頭）。名詞は folder と file の 2 つだけ。
4. **カーム・ローカルファースト・Notion 化しない**（requirements §9）。機能は「1 実装を足すだけ」の抽象（`SearchIndex` / `VelqPlugin` / `Exporter`）の上に載せる。
5. **UI に git 語彙を出さない**（plan §10）— 日本語化する場合も同じ（「コミット」「ブランチ」「マージ」は出さない。「保存履歴」「版」「変更点」）。

表記: **効果** = どの柱に効くか / **規模** = S（~1日）・M（~1週）・L（それ以上）。

## 3. 新規提案（文書にまだ無いもの）

### 3.1 `.velq` エコシステム — 「普及最優先」に直結

**E1. URL → `.velq`（ウェブクリッパー）** — 効果: .velq / 規模: S–M
コマンドパレットか File メニューに「Save web page as .velq…」。URL を貼る → reqwest で HTML 取得 →
既存の bundler パイプライン（CDN 取得・ハッシュ化・書換）→ `Documents/Velq` に保存。
`manifest.sourceUrl` は最初から仕様にある（velq-format.md）＝この機能は当初から想定されていた。
SingleFile が証明した需要を「開くのも安全（隔離ビューワ）」込みで取れるのは Velq だけ。
ウィンドウへの URL ドロップ（既存 `useFileDrop` の拡張）も同じ導線に。

**E2. macOS Quick Look 拡張（Finder で Space → `.velq` プレビュー）** — 効果: .velq / 規模: L
`.velq` がアプリの外（Finder・メール添付・Slack 保存物）でも一級市民になる。フォーマット普及の
インフラ投資。Tauri 本体とは別の Swift extension バンドルになるため独立トラックで。Windows の
プレビューハンドラは後続。

**E3. `velq` CLI（`pack` / `unpack` / `validate` / `open`）** — 効果: .velq / 規模: S — **⏸ 見送り（2026-07-06 ユーザー判断「CLI 入らないかも」。需要が出たら再検討）**
crates は全部 headless（architecture.md）なので、薄い bin crate を 1 個足すだけ。CI で HTML 成果物を
`.velq` 化する、スクリプトで一括変換する、といった開発者導線がフォーマットの裾野を広げる。
`cargo install velq` / Homebrew 配布。

**E4. ブラウザで開く `.velq`（JS/WASM ビューワ + velq.sh に「ドロップで開く」）** — 効果: .velq / 規模: M–L
ZIP + HTML という素性の良さを活かし、velq.sh 上に「.velq をここにドロップ」→ その場で閲覧
（sandboxed iframe + CSP、通信なし）を置く。**アプリを入れていない人も開ける**ことはフォーマットの
生存保証であり、Phase 2 の共有リンク（requirements §3.1）のビューワ実装をそのまま先取りする。

**E5. ビューワの読書体験** — 効果: .velq / 規模: M
現状の隔離ビューワは「安全だが素っ気ない」。①ウィンドウタイトルに `manifest.title`、②⌘F ページ内検索、
③ズーム、④印刷 / PDF 書き出し、⑤「Velq に取り込んで編集」（unpack → vault へコピー）、
⑥「この文書はオフライン・隔離表示です」の静かなトラストバッジ（初回のみ）。
※ viewer は capability ゼロを維持し、チェンジはネイティブメニュー / ウィンドウ側で行う（§7 の隔離を壊さない）。

**E6. パッケージ結果の見える化** — 効果: .velq / 規模: S
`BundleReport.failed` は現状ほぼトーストのみ。失敗 URL の一覧パネル＋リトライ＋成功時の
「✓ 完全オフラインで動きます」確認ランプ。オフライン完全性は .velq の売りなので、**保証されたことを
見せる**ところまでが機能。

**E7. 再パッケージの紐付け（`.velq` 増殖の解消）** — 効果: .velq / 規模: S–M
現状 `unique_velq` により、同じ HTML を開き直すたび `page 2.velq`, `page 3.velq`… が増える
（[bundle.rs](../apps/desktop/src-tauri/src/commands/bundle.rs) の `unique_velq`）。ソース → 出力の対応を
記憶して**同じ `.velq` を更新**し、「Re-package」コマンドを明示的に持つ。実害のある UX 課題なので優先度高め。

### 3.2 書く体験

**W1. 画像のペースト & ドラッグ挿入** — 効果: エディタ / 規模: M（トップ 5 #2）— **✅ 完了（M33。⌘V で attachments/ に保存+リンク挿入、エディタへの画像ファイルドロップも同挙動）**
クリップボード画像 / ドラッグ画像を `assets/`（またはドキュメント隣接フォルダ、設定制）へ保存 →
相対リンクを挿入 → Live で widget 描画。バイナリは履歴から除外済み（plan §10）なので設計と整合。
スクリーンショット貼り付けは技術文書・ブログ執筆の生命線で、これが無いと Obsidian から移れない。

**W2. 内部リンクの補完 + ドラッグでリンク挿入** — 効果: エディタ×ファイル管理 / 規模: S–M — **✅ 完了（M33。`[[` で補完→プレーンな相対リンク挿入。ドラッグ挿入は次回）**
`](` 入力で vault 内ファイルのパス補完。ツリー / リストからエディタへファイルをドラッグ → 相対 Markdown
リンク挿入。**グラフビューはやらない**（spec の明示アンチパターン）——あくまで「ただのファイルへの、
ただの相対リンク」で、ローカルファーストの範囲に留める。

**W3. スプリット（タブを左右に並べる）+ ピン留めタブ** — 効果: エディタ / 規模: M–L — **✅ 完了（M33。タブ右クリック→「右に並べて開く」/ピン留め。第2ペインも通常の編集・自動保存対象）**
spec §1 の「参照ドキュメントと下書きを並べる」。タブ右クリック→ Split right と、タブの画面端ドラッグ。
ピン留めで「close others」から保護。CM6 ラッパは view 単位で独立しているので構造的な障害はない。

**W4. スペルチェック切替** — 効果: エディタ / 規模: S — **✅ 完了（M33。設定→エディタ、CM と見たまま編集の両方に適用）**
WebView ネイティブの spellcheck 属性のトグルを Settings に。英語で書く人向けの安価な一手。

**W5. セッション復元** — 効果: エディタ / 規模: S–M — **✅ 完了（M33。タブ・アクティブ・ピン・分割・タブ別モードを復元。スクロール位置は未対応）**
再起動時に開いていたタブ・アクティブタブ・スクロール位置を復元（現状は lastVault と Recents のみ）。
「昨日の続きから」が北極星のカームさに直結する。

**直接操作で編集する（プレビュー直編集 → 要素選択 → AI）** — 編集の *重さ* で入口を 3 段に分ける発想。
① ちょっとした文言 = レンダリング結果の上で直接（W6）/ ② 要素という塊の単位 = 選択して操作（W7）/
③ それを超える変更・デザイン = AI かコード（W8）。3 つとも「描画された DOM ↔ ソースの**位置対応**」という
1 つの基盤に乗るので、W6 で基盤を作り W7・W8 がその上に乗る順で積むのが最短。着想は 2026-07-03 の
ユーザー議論（HTML の文言をちょろっと直したい／div を選んで AI にやらせたい）。

**W6. プレビュー結果の上で直接、文言を直す（HTML 先行）** — 効果: エディタ / 規模: M — **✅ 完了（M25＋M27。文言に留まらず構造編集・書式・全面「見たまま」モードまで）**
「デザインや構造の大改造はコードで、ちょっとした言い回しの手直しはレンダリング結果の上で」——編集の *重さ* で
入口を分ける、その軽い側。プレビュー iframe を `contenteditable` にし、テキストノードの編集を**元ソースの
正しい位置へ書き戻す**。核心は DOM ↔ ソースの**位置対応（source position mapping）**で、素朴な文字列置換だと
同じ語（"OK" が 2 箇所…）を巻き込むため、パース時に各テキストノードへソース offset を持たせて対応づける。
プレビューは既に**同一オリジン iframe**（`sandbox="allow-same-origin"`、
[PreviewPane.tsx](../apps/desktop/src/preview/PreviewPane.tsx)）で親から `contentDocument` を直接触れるので
土台は済んでいる。**HTML を先行**（DOM とソースが近い）、Markdown は逆写像（描画後 DOM→MD）が重いので後段。
これは Notion 型ブロックエディタの導入ではない（§5 のアンチパターンを踏まない）——**既存文書のテキストを
その場で直すだけ**。前提として HTML が編集対象として開けること（現状 `autoPackageHtml` 既定 true でファイルから
開くと `.velq` 化に逸れる → 開く導線の分岐が先。下記 §6）。

**W7. 要素選択モード（インスペクタ的に「この塊」を掴む）** — 効果: エディタ / 規模: S–M — **✅ 完了（M33。見たまま編集中に ⌥クリックで選択→削除/複製。DOM 主権の直列化に乗るので位置対応は不要だった）**
プレビュー上でホバーすると要素が薄くハイライトし、クリックで確定（DevTools の要素選択に近い体験）。掴んだ
要素 → ソース範囲の対応は **W6 の位置対応基盤をそのまま再利用**するので、W6 を先に作ると安く付く。選択それ
自体が「この段落だけコピー / 消す / 複製」のローカル操作の起点にもなり、次の W8（AI）の入力にもなる。同一
オリジン iframe なので選択 UI は親から素直に実装できる。

**W8. 掴んだ要素を AI に渡して直す（Phase 2 疎結合／本体は "口" だけ）** — 効果: エディタ×Phase 2 / 規模: 本体 S＋別リポ L
「文言直しを超える変更は AI に」。W7 で掴んだ要素（`outerHTML` ＋ソース範囲）を AI に渡し、返ってきた書き換えを
**非破壊で適用＋差分レビュー**（保存履歴の柱に乗せる）。ただし AI 実体・API キー・プロンプトは**本体に入れない**
——plan §18 の通り Phase 2・別リポ・プロプラで、プラグイン API か Tauri command/HTTP 経由の**疎結合**
（[plan.md](../plan.md) L455・L704／AGPL 伝播回避）。本体の責務は「選択コンテキストを提供する口」と「返差分を
適用・レビューする受け皿」に限る。ここを混ぜると設計の根幹（ライセンス分離）が壊れるので、一気にやらず
W6 → W7 →（口）の順で積む。

### 3.3 ファイルマネージャ & 検索

**F1. 全文検索（Turso FTS）+ 検索結果画面** — 効果: ファイル管理 / 規模: M–L（トップ 5 #5）
`SearchIndex` trait の第 2 実装として計画通り Turso を導入（velq-search / メモリの M9 決定とも整合:
swap ではなく段階導入）。⌘⇧F で結果リスト（ヒット行スニペット＋ハイライト＋クリックでジャンプ）。
インデックス更新は既存の fs watcher イベントに乗せる。
※ Turso の FTS が beta 制約に当たった場合は plan §16-R1 の通り**ユーザーに相談**（勝手に別 DB へ逃げない）。

**F2. タグ MVP（frontmatter ベース）** — 効果: ファイル管理 / 規模: M
`tags:` frontmatter を読み取り、サイドバー TAGS セクション（spec §2.2 の色ドット）＋クリックで横断
フィルタ＋右クリック「Add tag ▸」（frontmatter を書き換え）。**DB を真実にしない**（ファイルが真実、
ローカルファースト）。Bear/Ulysses が証明した、フォルダを壊さない第 2 軸。

**F3. 「最近削除した項目」ビュー + 復元** — 効果: ファイル管理×保存履歴 / 規模: M
削除は OS ゴミ箱行きだが、復元導線がアプリ内に無い。保存履歴バックエンドには削除前の内容が残っている
ので、「最近削除」スマートビュー（spec §2.2 の Trash）から**一クリック復元**できる。柱 1 と柱 2 の
交差点で、他エディタに無い安心感になる。

**F4. アイコングリッドの熟成** — 効果: ファイル管理 / 規模: S–M（個別に小粒）
M21 で新設した初心者ファーストのグリッドに、①ソート切替（名前 / 更新日 / 作成日 / 手動）、
②アイコンサイズスライダー、③フォルダカラー / 絵文字（小さくキュレートした選択肢、spec §2.2）。
Spacedrive / Files app 比較の次の一歩。

### 3.4 日本語と国際化（作者・初期ユーザー層に直結）

**J1. 文字数カウント** — 効果: エディタ / 規模: S — ✅ **M22 実装済み**
ステータスバーは `N words` のみだった（[StatusBar.tsx](../apps/desktop/src/shell/StatusBar.tsx)）。
**日本語では word count は無意味**なので、CJK 主体（Han/かな ≥20%）なら「1,204 文字」、それ以外は「340 words」を
自動表示（桁区切り付き、もう一方はホバーで表示）。`countChars` は空白・改行・全角スペースを除いたコードポイント数で、
原稿用紙の数え方と一致。ヘルパは `util/text.ts`（12 ケースの単体テスト）。

**J2. UI の日本語化（i18n 基盤 + ja）** — 効果: 全体 / 規模: M — ✅ **M23 実装済み**
薄い辞書ベースの i18n（`src/i18n/`：`en.ts` を真実、`ja.ts` は型で全キー一致を強制）。設定に「言語」
セレクタ（システム / English / 日本語、既定＝OS 言語に追従）。**禁止語ルールを日本語にも適用**
（「コミット」「リポジトリ」「マージ」を訳語にも出さない——「版」「保存履歴」「変更点」「元に戻す」）。
ウェブビュー UI 全域＋ネイティブメニューを **再起動なしでライブ切替**（`apply_menu_language` でメニュー再構築）。
言語追加は辞書ファイル 1 個で済む。残（英語のまま）: Rust 側のコマンドエラー文言・`velq-vcs` の版サマリ
（「3 added · 1 removed」＝カウントの DTO 化が必要）・プラグイン説明文・初期 Welcome.md／既定ファイル名。

**J3. 日本語全文検索（lindera）** — 効果: ファイル管理 / 規模: M
requirements §2.3 の将来項目。F1 の FTS が入った後、トークナイザを差し替える第 2 段として。

**J4. 和文タイポグラフィ** — 効果: エディタ / 規模: S
和文時の measure（全角 ~35–40 字）、行間 1.8 寄り、和文フォントスタック（Hiragino / Noto Sans JP）を
プリセットに。spec §3.4 の欧文前提を日本語に拡張する。
